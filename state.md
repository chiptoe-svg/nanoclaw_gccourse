## Goal

NanoClaw is a self-hosted personal-Claude assistant. The Clemson install (Mac Studio at `130.127.162.180:3002`) pairs per-person agents over a shared messaging surface with a playground UI for chat and admin. **As of 2026-06-07 the classroom is a PILOT/test with pilot users — breakable, not live production** (the prior "production, not dev" framing is paused until the owner says otherwise; no real FERPA data expected during the pilot).

**Direction (revised 2026-06-08):** this is a **group-agent platform**, not a classroom app — controlled, individuated agent access for a defined set of people, with "classroom" as one of ~5 scenarios (department, agent-optimization, +2 coming). **Model: ONE codebase + in-tree scenario profiles** under `src/scenarios/<name>/`, selected by config. This supersedes both prior attempts: lean-trunk+branch-install (too much sync ceremony) and fold-into-classroom-app (too narrow). Each scenario's code is tiny (~90% of "classroom" code is the general platform), so in-tree profiles beat branch-install ceremony. Separate installs run the same repo with different scenario config + data; platform features reach all installs on `git pull`. See `plans/group-agent-platform.md`. (The earlier `controlled-access` branch-extraction plan and the `classroom` sibling branch + `/add-classroom*` skills are superseded; branch/skills slated for retirement.)

## Invariants (don't break)

These are load-bearing. If a commit would violate one, surface it before writing the code.

- **The two-DB session model is the only IO surface between host and container.** `inbound.db` is host-write / container-read. `outbound.db` is container-write / host-read. There is no IPC, no stdin piping, no file-watch. See `container/agent-runner/src/db/connection.ts` and `docs/db-session.md`.
- **`outbound.db` is container-owned; the host opens it readonly except for `host-sweep.ts`'s claim cleanup and `session-manager.ts`'s `writeOutboundDirect` (RW opener).** Any new host-side write to outbound.db requires using `openOutboundDbRw` AND must be crash-safe (stale `-journal` files from a SIGKILL leave the file in a "needs recovery" state that breaks subsequent readonly reads — `recoverStaleOutboundJournals` runs at startup to clean them).
- **Session DBs use `journal_mode=DELETE`, not WAL.** Cross-mount visibility depends on this. The pragma is in `container/agent-runner/src/db/connection.ts` and in the host openers in `src/db/session-db.ts`.
- **Container-side bun:sqlite needs `$name` in JS keys** (`db.prepare(...).run({ $id: foo })`); host-side better-sqlite3 auto-strips. Mixing the conventions silently breaks named-param SQL.
- **Trunk is infrastructure-only.** Channel adapters (Discord, Slack, Telegram, …) live on the `channels` branch; non-default agent providers live on the `providers` branch. Each is installed by a skill. Don't add channel-specific or provider-specific code to trunk. See `CLAUDE.md` § 5.
- **pnpm `minimumReleaseAge: 4320` (3 days) is enforced.** New deps wait 3 days from publish. `minimumReleaseAgeExclude` and `onlyBuiltDependencies` additions require explicit human approval — never bypass.
- **Credentials never sit in container env vars or chat context.** Containers get placeholder credentials; the host credential-proxy at `127.0.0.1:3001` substitutes real keys/tokens at request time. The 4 Anthropic-OAuth landmines (sk-ant-oat- prefix, `anthropic-beta: oauth-2025-04-20`, Claude Code preamble, model.baseUrl override) are all live and must be preserved. See `src/credential-proxy.ts`.
- **`agent_groups.agent_provider`, `container_configs.provider`, and `sessions.agent_provider` must all be `'pi'` post-Phase-D.** Migration 023 backfilled this; `scripts/migrate-to-pi.ts` is the recovery tool. Drift between the three columns triggers the wrong auth-dir mount and was the bug `migrate-to-pi.ts` was written to fix.
- **MCP tool handlers (`container/agent-runner/src/mcp-tools/`) run in a SEPARATE PROCESS from the poll-loop.** Module-level state in `current-batch.ts` is NOT shared. Anything needing to flow from poll-loop to MCP tools must go through the DB (or the spawned tool's args), not module globals. The messages_out usage backfill (commit `b204567`) uses a seq lower bound for exactly this reason.
- **Container `--rm` flag means container stdout/stderr is lost on exit.** If an agent silently failed inside the container, there's no persistent log. Make session-DB writes the source of truth for anything diagnosed post-hoc.
