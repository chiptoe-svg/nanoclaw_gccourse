NanoClaw — How It Works

Behind the scenes: components, data flow, and the security model. Generated 2026-06-07. Companion to the diagram view in docs/architecture.html, the prose in docs/architecture.md, and state.md.

NanoClaw is a self-hosted personal-Claude assistant. A single Node host process orchestrates one short-lived agent container per conversation session. The host and containers never share memory, sockets, or files directly — they communicate through two SQLite files per session, and nothing else. Real API credentials never enter a container; a local proxy substitutes them on the wire.

1 · The big picture

Everything is a message. A platform message arrives through a channel adapter, is routed to the right session, written into that session's inbound database, and a container is woken. The agent inside the container polls its inbound DB, calls the model, and writes replies to its outbound DB. The host polls the outbound DB and delivers replies back through the same adapter.

External

Chat platforms (Discord, Slack, Telegram, …) & the web playground. LLM APIs (Anthropic, OpenAI, local OMLX, Clemson RCD).

Host process (Node)

Channel adapters · router · delivery · host-sweep · session-manager · container-runner · credential-proxy · central DB.

Per-session container (Bun)

The agent-runner: poll loop → pi harness → model call → MCP tools (send_message, send_file, knowledge_search, …).
platform / playground │ inbound ▼ ┌──────────────────────── HOST (Node) ────────────────────────┐ │ channel adapter → router → inbound.db → wake container │ │ ▲ │ │ adapter ← delivery ← outbound.db ←────────────┘ (poll) │ │ ▲ │ │ credential-proxy (127.0.0.1:3001) ──► LLM APIs │ └─────────────────────────│──────────────────▲────────────────┘ mounts inbound.db (ro) │ outbound.db (rw) │ placeholder creds ▼ │ ┌──────────────── CONTAINER (Bun, --rm) ──────────────────────┐ │ poll inbound.db → pi harness → model via proxy → write │ │ outbound.db; MCP tools for messages, files, retrieval │ └─────────────────────────────────────────────────────────────┘
The container only ever talks to two local SQLite files and the local proxy. It has no platform tokens, no real API keys, and no direct network trust. That isolation is the backbone of the security model (§5, §10).

2 · Entity model

Routing and permissions are built on a small set of entities in the central DB (data/v2.db):

users ("<channel>:<handle>", kind, display_name) │ ├── user_roles owner | admin (global or scoped to an agent group) ├── agent_group_members unprivileged access gate └── user_dms cold-DM cache agent_groups (workspace, memory, CLAUDE.md, personality, container config) ▲ many-to-many via messaging_group_agents (session_mode, triggers) ▼ messaging_groups (one chat/channel on one platform; unknown_sender_policy) sessions = agent_group × messaging_group × thread_id → one container
  • Privilege is user-level (owner / admin), not group-level. An owner can do anything; an admin may be scoped to specific agent groups; members get access without privilege.
  • A session is the unit of execution — each one gets its own container and its own pair of databases.
  • Three isolation levels (agent-shared, shared, separate agents) control how much state different chats share. See docs/isolation-model.md.

3 · A message, end to end

#WhereWhat happens
1platformA user sends a message; the channel adapter receives it and calls the router.
2routerResolves user → messaging group → agent group → session. Applies unknown_sender_policy and the command gate (admin commands checked against user_roles).
3inbound.dbThe message is written to messages_in with routing + destinations. Host uses even seq numbers.
4container-runnerIf no container is running for the session, one is spawned with the two DBs + outbox mounted and the proxy env vars injected.
5poll loopThe agent-runner polls inbound.db, sees the new row, builds the prompt, and runs the pi harness.
6pi → proxyThe model call goes to http://host.docker.internal:3001 with a placeholder credential. The proxy swaps in the real key/token and forwards to the LLM API.
7MCP toolsThe agent may call tools: send_message, send_file, knowledge_search, scheduling, agent-to-agent. Tool handlers run in a separate process from the poll loop — they communicate only through the DB.
8outbound.dbReplies are written to messages_out (container uses odd seq). Files land in the session outbox/.
9deliveryThe host polls outbound.db, reads any outbox files, and delivers through the originating adapter. System actions (schedule, approvals) are handled here.
10host-sweepEvery 60s: syncs processing_ack, detects stale/crashed containers, wakes due (scheduled) messages, handles recurrence.

4 · The two-DB session model

Each session has two SQLite files under data/v2-sessions/<session>/:

inbound.db

Host writes, container reads. messages_in, routing, destinations, pending_questions, processing_ack.

outbound.db

Container writes, host reads. messages_out, session_state. Host opens it read-only except for sweep claim-cleanup.

Invariants that must never break

  • Exactly one writer per file. No cross-mount lock contention; this is why the split exists.
  • journal_mode = DELETE, not WAL. Cross-mount visibility (host ↔ container over a bind mount) depends on it. WAL's shared-memory file isn't visible across the mount boundary.
  • Seq parity: host writes even seq, container writes odd — so the two writers never collide on an id.
  • Stale-journal recovery: a SIGKILL mid-write can leave a -journal that makes read-only opens fail; the host runs recoverStaleOutboundJournals at startup to clean them.
  • Heartbeat is a file touch at /workspace/.heartbeat, not a DB row — keeps liveness off the IO surface.

A third database, the central data/v2.db, holds everything that isn't per-session: users, roles, agent/messaging groups, wiring, approvals, schema version, migrations.

5 · Credentials & the proxy — the heart of the security model

Real secrets live only on the host, in .env and the OS keychain. Containers receive placeholder values. The local credential proxy at 127.0.0.1:3001 substitutes the real credential at request time and forwards upstream.

container sets: ANTHROPIC_BASE_URL = http://host.docker.internal:3001 ANTHROPIC_API_KEY = "placeholder" (or CLAUDE_CODE_OAUTH_TOKEN) OPENAI_BASE_URL = .../3001/openai/v1 → multiplexed by path prefix container ──(placeholder)──► proxy ──(real key / OAuth bearer)──► api.anthropic.com │ api.openai.com ├─ rewrites x-api-key / Authorization chatgpt.com (codex OAuth) ├─ refreshes OAuth tokens on expiry OMLX / Clemson RCD └─ per-student / class-pool resolver
  • Nothing sensitive is ever in container env or chat context. Even the agent can't read the real key — it only holds the placeholder.
  • Provider multiplexing is by path prefix: /openai/, /openai-platform/, /omlx/, /clemson/, default = Anthropic.
  • OAuth landmines (load-bearing, see credential-proxy.ts): the sk-ant-oat- prefix, the anthropic-beta: oauth-2025-04-20 header, the Claude Code system-prompt preamble, and a model.baseUrl override must all be present in OAuth mode.
  • Per-student / class-pool resolver (classroom): the proxy can resolve a student's own connected credential first, then fall back to the instructor's class pool, then deny — without the student ever seeing a key.

Recent hardening (2026-06-07)

  • Keychain reads are non-blocking with a timeout (no event-loop stall under load).
  • Upstream requests have a 120s timeout and are torn down if the container disconnects (no leaked sockets / wasted quota).
  • The per-session payload-log handle map is LRU-capped (no FD leak across a long day).

6 · Container lifecycle

  • Spawn on demand: container-runner starts a container per agent group/session, mounting the session DBs + outbox and injecting proxy env vars. Runtime is Docker or Apple Containers (container-runtime.ts).
  • Liveness: the agent touches /workspace/.heartbeat; the host sweep detects stale containers.
  • --rm: containers are removed on exit, so stdout/stderr is lost. Anything needed for post-hoc diagnosis must be written to the session DB — the DB is the source of truth.
  • Orphan cleanup reaps containers left behind by a host crash.

7 · Trunk, channels & providers (small-trunk-with-skills)

Trunk (the main branch) is meant to be only the infrastructure every install needs. Optional capabilities live on long-lived sibling branches and are copied in by /add-<x> skills:

trunk (main)

routing, two-DB model, container orchestration, credential proxy, permissions, the playground shell.

channels branch

Discord, Slack, Telegram, WhatsApp, Teams, Linear, GitHub, iMessage, … — installed via /add-<channel>.

providers branch

Non-default agent providers (e.g. OpenCode) — installed via /add-opencode.
Known debt + planned re-architecture: the "classroom" code (~40 files) is really a general controlled-access agent layer (per-user provisioning, end-user onboarding/auth, BYO + pooled credentials, member policy) plus a thin teaching veneer plus a Google integration. With three real consumers (classroom, department agents, an agent-optimization class) it's being extracted as a reusable core + per-profile skills. Plan: plans/controlled-access-core.md. It works today; this is post-term hygiene, not a bug.

8 · Knowledge & retrieval (RAG)

Documents are ingested into per-corpus indexes the agent can search at conversation time.

raw files ─► extract text ─► chunk (sentence | fixed, size + overlap) │ ┌──────────┴───────────┐ ▼ ▼ BM25 index dense index (SQLite FTS5) (OpenAI embeddings → cosine) │ │ └─────── query ─────────┘ Retrieval tab: BM25 / dense / hybrid blend
Capability seam to know about: the host-side Retrieval/Benchmark tabs support BM25, dense, and hybrid — but the agent's knowledge_search MCP tool is BM25-only. A dense-only corpus is therefore invisible to the agent at chat time. A planned "pipeline explorer" feature (docs/superpowers/specs/2026-06-06-pipeline-explorer-knobs-design.md) surfaces this and lets students compare strategies live.

9 · The playground UI

A web app (src/channels/playground/) that is itself a channel adapter. Tabs: homechatpersonaskillsmodelsagentssourcesretrievalbenchmarks

  • Chat rides the same two-DB pipeline as any other channel; replies stream to the browser over SSE, with a /recent backfill so a dropped connection doesn't lose a reply.
  • Auth: magic-link sessions in production. A dev bypass mode exists for local setup but is now fenced (see §10).
  • Per-folder authorization: read/mutation of an agent group's drafts, persona, skills, usage, and files is gated by canReadDraft / checkDraftMutation against the caller's identity.

10 · Security strategy (summary)

LayerStrategy
Credential isolationReal keys/tokens never leave the host. Containers hold placeholders; the proxy substitutes on the wire. Secrets are never in container env, chat context, or logs.
Execution isolationContainers are sandboxed and talk only to two local SQLite files + the local proxy. No platform tokens, no direct outbound trust. --rm so nothing persists.
AuthenticationPlayground uses magic-link sessions. A dev auth-bypass mode exists; a startup guardrail now refuses to run a bypass on a non-loopback bind, so it can't be left on accidentally when exposed.
AuthorizationUser-level roles (owner/admin/member) + per-agent-group access checks. Per-folder read/mutation gates on the playground API. Admin commands are checked router-side against user_roles.
Input safetyPath-traversal guards on every file route (isSafeAttachmentName); parameterized SQL throughout; uploaded filenames sanitized + validated.
Supply chainpnpm minimumReleaseAge (3 days) so new dependency versions must age before they resolve; build-script execution is allow-listed.
Resource safetyUpstream timeouts + disconnect teardown in the proxy; LRU-capped handle maps; TTL sweeps for staged files; bounded embedding concurrency.
