# Phase 7D — Agent `knowledge_search` MCP Tool

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Expose the Phase 7A BM25/FTS5 knowledge corpora to agents inside their container via a new `knowledge_search` MCP tool. Agents can query one or all ready corpora and get ranked results back as readable text.

**Architecture:** A single new file `container/agent-runner/src/mcp-tools/knowledge.ts` exports named tool objects and calls `registerTools([...])` at module scope. It reads corpora from `/workspace/agent/knowledge/corpora/` (overridable via `AGENT_DIR` env var for tests), discovers ready corpora by reading `meta.json`, opens each `bm25.db` read-only using `bun:sqlite`, runs a `MATCH` query, collects all results, re-ranks by BM25 score (higher absolute value = more relevant), and returns top-k formatted as plain text. The file is imported in `index.ts` for its side-effect. No new runtime dependencies — `bun:sqlite` is part of the Bun runtime.

**Tech Stack:** Bun runtime, `bun:sqlite` (built-in — no install needed), `bun:test` for tests. FTS5 `bm25()` returns negative values; absolute value is used for ranking and display (consistent with the host-side `store-bm25.ts` convention).

function makeCorpus(
  id: string,
  name: string,
  status: string,
  chunks: Array<{ id: string; source: string; chunk_index: number; text: string }>,
): void {
  const dir = path.join(corporaDir(), id);

    id,
    name,
    status,

    chunkStrategy: 'sentence',

      id UNINDEXED,

      source UNINDEXED,
      chunk_index UNINDEXED,

      insert.run({ $id: c.id, $corpus_id: id, $source: c.source, $chunk_index: c.chunk_index, $text: c.text });
    }
    db.close();
  }
}

#### Step 2: Write the implementation

```typescript
// container/agent-runner/src/mcp-tools/knowledge.ts

/**
 * knowledge_search MCP tool.
 *
 * Searches ready BM25/FTS5 knowledge corpora mounted at
 * /workspace/agent/knowledge/corpora/ (override via AGENT_DIR env var).
 *
 * Each corpus has a meta.json (status, name, id) and a bm25.db (FTS5 virtual
 * table). This tool opens each bm25.db read-only with bun:sqlite, runs a
 * MATCH query, collects results from all ready corpora, re-ranks by BM25
 * score, and returns the top-k as formatted text.
 *
 * BM25 score sign: bm25() returns negative values — lower is more relevant.
 * We use Math.abs(score) for ranking (descending) and display, consistent
 * with the host-side store-bm25.ts convention.
 */
import { Database } from 'bun:sqlite';
import fs from 'fs';
import path from 'path';

import { registerTools } from './server.js';
import type { McpToolDefinition } from './types.js';
import type { CallToolResult } from '@modelcontextprotocol/sdk/types.js';

function agentDir(): string {
  return process.env.AGENT_DIR ?? '/workspace/agent';
}

function corporaDir(): string {
  return path.join(agentDir(), 'knowledge', 'corpora');
}

interface CorpusMeta {
  id: string;
  name: string;
  status: string;
}

interface QueryRow {
  id: string;
  corpus_id: string;
  source: string;
  chunk_index: number;
  text: string;
  score: number; // raw bm25() value — negative; abs = relevance
}

interface RankedResult {
  corpusName: string;
  source: string;
  chunkIndex: number;
  score: number; // absolute value
  text: string;
}

function readMeta(corpusPath: string): CorpusMeta | null {
  try {
    const raw = fs.readFileSync(path.join(corpusPath, 'meta.json'), 'utf8');
    return JSON.parse(raw) as CorpusMeta;
  } catch {
    return null;
  }
}

function listReadyCorpora(): Array<{ meta: CorpusMeta; dir: string }> {
  const dir = corporaDir();
  if (!fs.existsSync(dir)) return [];
  return fs
    .readdirSync(dir, { withFileTypes: true })
    .filter((e) => e.isDirectory())
    .flatMap((e) => {
      const corpusPath = path.join(dir, e.name);
      const meta = readMeta(corpusPath);
      if (!meta || meta.status !== 'ready') return [];
      return [{ meta, dir: corpusPath }];
    });
}

function searchCorpus(dir: string, query: string, k: number): QueryRow[] {
  const dbPath = path.join(dir, 'bm25.db');
  if (!fs.existsSync(dbPath)) return [];
  const db = new Database(dbPath, { readonly: true });
  try {
    const rows = db
      .prepare(
        `SELECT id, corpus_id, source, chunk_index, text, bm25(chunks) AS score
         FROM chunks
         WHERE chunks MATCH ?
         ORDER BY score
         LIMIT ?`,
      )
      .all(query, k) as QueryRow[];
    return rows;
  } finally {
    db.close();
  }
}

function ok(text: string): CallToolResult {
  return { content: [{ type: 'text', text }] };
}

function mkErr(text: string): CallToolResult {
  return { content: [{ type: 'text', text }], isError: true };
}

function formatResults(results: RankedResult[]): string {
  return results
    .map(
      (r, i) =>
        `[Result ${i + 1}]\n` +
        `Corpus: ${r.corpusName}\n` +
        `Source: ${r.source} (chunk ${r.chunkIndex})\n` +
        `Score: ${r.score.toFixed(4)}\n` +
        `${r.text}`,
    )
    .join('\n\n---\n\n');
}

export const knowledgeSearch: McpToolDefinition = {
  tool: {
    name: 'knowledge_search',
    description:
      "Search the agent group's knowledge corpora using BM25 full-text search. " +
      'Returns the top-k most relevant chunks ranked by BM25 score. ' +
      'Search all ready corpora by default, or limit to one with corpus_id.',
    inputSchema: {
      type: 'object',
      properties: {
        query: {
          type: 'string',
          description: 'Full-text search query (FTS5 MATCH syntax). Keep it simple — keywords or phrases.',
        },
        k: {
          type: 'number',
          description: 'Maximum number of results to return (default: 5).',
        },
        corpus_id: {
          type: 'string',
          description: 'If given, search only this corpus (by id). Omit to search all ready corpora.',
        },
      },
      required: ['query'],
    },
  },

  handler: async (args: Record<string, unknown>): Promise<CallToolResult> => {
    const query = String(args.query ?? '').trim();
    const k = typeof args.k === 'number' && args.k > 0 ? Math.floor(args.k) : 5;
    const corpusId = typeof args.corpus_id === 'string' ? args.corpus_id.trim() : undefined;

    if (!query) {
      return mkErr('query is required and must be a non-empty string.');
    }

    // Targeted search: corpus_id specified
    if (corpusId) {
      const dir = path.join(corporaDir(), corpusId);
      if (!fs.existsSync(dir)) {
        return mkErr(`Corpus "${corpusId}" not found.`);
      }
      const meta = readMeta(dir);
      if (!meta) {
        return mkErr(`Corpus "${corpusId}" has no readable meta.json.`);
      }
      if (meta.status !== 'ready') {
        return mkErr(
          `Corpus "${corpusId}" is not ready (status: ${meta.status}). Build it in the Sources tab first.`,
        );
      }

      let rows: QueryRow[];
      try {
        rows = searchCorpus(dir, query, k);
      } catch (e) {
        return mkErr(`Invalid search query: ${e instanceof Error ? e.message : String(e)}`);
      }

      if (rows.length === 0) return ok('No relevant chunks found.');

      const results: RankedResult[] = rows.map((r) => ({
        corpusName: meta.name,
        source: r.source,
        chunkIndex: r.chunk_index,
        score: Math.abs(r.score),
        text: r.text,
      }));

      return ok(formatResults(results));
    }

    // All-corpora search
    const ready = listReadyCorpora();
    if (ready.length === 0) {
      return ok('No ready knowledge corpora found. Build one in the Sources tab.');
    }

    // Collect top-k from each corpus, then globally re-rank and slice to k.
    // Fetching k from each corpus before merge is a practical approximation —
    // no IDF normalization across corpora, but good enough for the RAG use case.
    const allResults: RankedResult[] = [];
    let ftsError: string | null = null;

    for (const { meta, dir } of ready) {
      let rows: QueryRow[];
      try {
        rows = searchCorpus(dir, query, k);
      } catch (e) {
        // FTS5 parse errors will be identical for every corpus — surface once.
        ftsError = e instanceof Error ? e.message : String(e);
        break;
      }
      for (const r of rows) {
        allResults.push({
          corpusName: meta.name,
          source: r.source,
          chunkIndex: r.chunk_index,
          score: Math.abs(r.score),
          text: r.text,
        });
      }
    }

    if (ftsError !== null) {
      return mkErr(`Invalid search query: ${ftsError}`);
    }

    if (allResults.length === 0) return ok('No relevant chunks found.');

    // Re-rank globally by descending score, take top-k.
    allResults.sort((a, b) => b.score - a.score);
    const topK = allResults.slice(0, k);

    return ok(formatResults(topK));
  },
};

registerTools([knowledgeSearch]);
```

## Implementation Notes

**BM25 score sign convention:** `bm25()` in SQLite FTS5 returns negative values (lower = more relevant). `Math.abs(score)` normalizes to a display-friendly positive number and sorts descending for top results. This matches the host-side `src/knowledge/stages/store-bm25.ts` convention.

**bun:sqlite named params:** Unlike `better-sqlite3` which strips the `$` prefix from JS object keys automatically, `bun:sqlite` requires the key in the JS object to include the `$` exactly as written in the SQL. Use `.run({ $id: val })` for `$id` in the SQL. Positional `?` params work the same way on both runtimes.

**FTS5 error surface:** SQLite throws synchronously when given malformed MATCH syntax (e.g., unmatched quotes). Wrap `db.prepare(...).all(...)` in try/catch. In the all-corpora path, break on the first FTS5 error and surface it — all corpora will reject the same invalid query, so processing the rest wastes cycles.

**k-per-corpus before merge:** Fetching up to k results per corpus before global merge is a practical approximation. In the worst case the globally top-k results are all from one corpus and valid results from others are discarded — acceptable for the RAG use case here, and avoids fetching unbounded rows per corpus.

**No new deps:** `bun:sqlite` is part of the Bun runtime. The `container/agent-runner/package.json` does not need any changes.

**`db.run()` vs `db.exec()` in bun:sqlite:** In the test fixture, use `db.run(sql)` for the `CREATE VIRTUAL TABLE` DDL — it's a single statement with no output. `db.exec()` in bun:sqlite is for multi-statement scripts. Prepared statement `.run()` is for parameterized DML.
