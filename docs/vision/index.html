NanoClaw
Classroom.

A small platform for teaching with agents.

A classroom tool that gives every student their own AI agent — separate workspace, separate credentials, isolated container. The instructor watches costs in real time, swaps providers when one acts up, and ships changes without reading vendor SDK release notes. Designed for about twenty-five concurrent sessions on a single Mac Studio.

Section 01 — Premise

What this is, and what it isn't.

NanoClaw is a fork of an existing personal-agent host, extended for classroom use. The underlying single-user architecture — per-session containers, two-DB session split, channel adapters, credential proxy — already isolates one user from the system. Multiplying that to a classroom turned out to be mostly mechanical; the work that actually mattered was everything a teacher specifically needs: roster, sign-in flow, per-student cost tracking, agent provisioning at scale, and a way to retract a misbehaving agent without rebooting the room.

This document covers what's built, what's coming, and the design choices behind both.

Three constraints shaped the design: it has to run a real classroom (around two dozen students working at the same time, instructor watching the room), the budget has to stay manageable (subscription tiers preferred over metered API where possible), and the architecture has to absorb new requirements — additional channels, providers, course formats — without a rewrite each time.

— Design brief, 2026-05-19

Section 02 — In the classroom

Built for teaching, organized around concepts.

Every tab in NanoClaw Classroom maps to something specific being taught. The design works backward from a teacher's intent — "today we're covering retrieval strategy" — to the surface that lets a student experience it. Each concept below has a clear home in the tool.

LLM vs agentthe conceptual pivot
Run the same prompt through direct mode (raw model) and agent mode (configured persona, skills, models). The trace panel surfaces what the agent adds — tool calls, memory, intermediate steps — so the difference between asking a chatbot and running an agent is visible, not abstract. Chat tab
Personashaping how an agent thinks
Edit Layer 3 of the agent's instructions and feel the difference in the next turn. Global → role → per-agent layers are visible side by side so students see exactly what they're changing and what's inherited. Persona tab
Skillswhat the agent knows how to do
Toggle capabilities on and off and re-run the same task. Skills are SKILL.md files the agent reads on demand; turning one on changes what the agent knows is available, not what it can technically execute. Skills tab
Tool wiringreal APIs, real outputs
Google Workspace (Docs / Sheets / Slides), web fetch, email, Telegram — wired through install skills at the platform level. Students don't install them, but they run real tasks against real Drive folders and watch the agent call tools live in the trace panel. Install layer · visible in Chat trace
Agent harnessmemory, context, reasoning
What an agent remembers, how much it can hold in mind at once, and when it decides to summarize. Three memory tiers (session, persistent CLAUDE.local.md, wiki), a live context-window utilization bar, the compaction trigger, and the reasoning-effort knob — all visible and adjustable. The "why did my agent forget?" lesson lives here. Harness tab · forthcoming
Knowledge retrievalwhat an agent knows beyond its training
Build corpora from PDFs, video, text, and data in the Sources lab bench, then design retrieve–rerank–prompt pipelines in the Retrieval lab bench. Compare strategies on the same query and see where each one breaks. Sources + Retrieval tabs · forthcoming
Evaluationmeasuring strategies against each other
Two strategies side by side on a shared question set. LLM-as-judge scoring plus human-spot-check; cost-normalized so quality-per-dollar settles out across runs. Eval tab · forthcoming
Take it homeportable agents
Export the tuned agent in one of four formats. The walkaway script stands it up on the participant's own infrastructure — persona, skills, and connections move off the classroom server cleanly. Export + walkaway · Phase 4
Cost monitoringinstructor view
Real-time per-student cost, model selection, enrollment status. The whole room visible at a glance, so an instructor catches a runaway thread or a provider going sideways before it disrupts class. Home tab

Why this works in practice

A few practical things matter when you're teaching with this. The whole install is self-contained — one bootstrap script on a fresh machine, no IT department in the loop, no cloud provisioning, no waiting on institutional accounts. A new course or section can be stood up in about an hour. Once it's running, students get going with a single email containing their playground URL plus the class passcode the instructor displays. Email magic-link and Telegram (for instructor admin) are wired through the same install — no per-student configuration. And because every session runs in its own container, students can break things in their own sandbox without affecting the room; an instructor recovers by killing one session, not by rebooting the install.

Section 05 — Architectural philosophy

Small trunk. Skills install everything else.

Every NanoClaw install gets the same trunk: a Node host that routes messages, a credential proxy that keeps real keys out of containers, a per-session container runtime, an entity model that ties users to messaging groups to agent groups to sessions. Trunk is infrastructure every install needs. Anything optional — channels, providers, classroom features, Google Workspace integration — lives on a long-lived branch and gets copied in by an install skill.

platform · agent · ui
Channel adapters
Telegram, playground (web), Discord, Slack, WhatsApp, iMessage, Teams, Resend, Matrix, gChat, GitHub, Linear, Webex, Signal, DeltaChat, WeChat. Each is a skill install.
trunk + skills
Router · entity model
Users → messaging groups → agent groups → sessions. Per-channel adapters land messages here. <message to="name"> routes across agent groups.
trunk
Credential proxy
Real keys live on host. Containers send placeholder credentials; proxy substitutes per-request. Multi-provider: Anthropic, OpenAI, Google Workspace, local MLX.
trunk
Per-session container
Apple Container (or Docker). Two-DB session split: inbound.db + outbound.db. One agent-runner instance per student session.
trunk
Agent providers
Claude Agent SDK · Codex app-server · OpenCode · Ollama · local MLX. Vendor SDK boundary lives here; the upper layers don't care which provider is active.
trunk + branches
Classroom features
Roster, login passcode, magic-link enrollment, per-student Google folders, per-student LLM auth, instructor home dashboard. /add-classroom, /add-classroom-gws.
skill installs
Workspace integrations
Google Drive Docs / Sheets / Slides / Calendar / Gmail via MCP relay. Bedrock-style ownership tag for shared-classroom mode.
skill installs
Admin tooling
Telegram-driven /auth, /model, /provider for live operator switching. Status bar, dashboard, install handoff.
skill installs

The dashed-border layers can be added or removed without trunk modification. The teal-border layers can't — they're load-bearing for every install. The contract between the two: skills may append to trunk barrels and call trunk-registered hooks, but trunk doesn't know skills exist. Net effect: the same trunk runs a one-person agent and a classroom of thirty.

Phase 3 ○ In progress

Triggered by a 2026-05-18 cost spike that turned out to be a trace-UI artifact (the "$1.10 yolo" was thread-cumulative, not turn cost). The bench suite gives us calibrated expectations before enrolling the next class, and regression detection across codex and SDK bumps. Five requests (Pong / Clock / Title / Lesson / Research) × six systems (claude sonnet+haiku, codex 5.4+5.4-mini, gemma+qwen MLX) × three reps. Three assessment layers: auto token/cost/latency, programmatic correctness per request, claude-haiku-judge quality rubric. Cost calculations apply provider-specific cache rates; context-window utilization is measured live during each run (peak input tokens / model max) so we can see which models flirt with overflow on multi-turn classroom workloads — that's the signal that catches local MLX models choking on long threads before a student does. Wire-level observability via a credential-proxy payload logger so we can answer "what bytes went up the wire?"

Section 07 — Cost in practice

What this actually costs.

The cost spike that triggered Phase 3 turned out to be a UI artifact, not a structural issue. Actual per-turn costs once cache warms up — measured on codex/gpt-5.4 with effort: low — settle at about half a cent per simple turn. Across a classroom-sized cohort × N turns per day × a semester, that puts the order of magnitude in single-digit dollars per student on metered API pricing. With subscription auth (ChatGPT Plus/Pro at $20–$200/mo, Claude Pro/Max similar) the ceiling drops further.

The classroom routes per-student by policy — instructor's class-pool credentials by default, student's own connected provider when authorized via the Home Providers card.

Section 08 — Open questions

What's still unresolved.

Trunk vs skill-installable boundary

Recent polish surfaced that this fork's trunk has accumulated classroom-specific code (Roster, X.7 Providers card, enrollment passcode). Project rule says trunk should be only what every install needs. Three options on the table; decision deferred until a real second classroom install or an upstream merge surfaces the cost.

Architectural simplification at the agent layer

The five hand-rolled provider implementations (claude / codex / opencode / ollama / local MLX) total roughly 2,000 lines and produce the messiest part of the codebase. An alternative library — multi-provider with subscription OAuth, in-process agent loop, built-in cost tracking — would collapse them into one provider. Best evaluated in a separate install before any production migration.

Phase 14 GCP unblock

Per-person Google Workspace OAuth blocked on five minutes of GCP Console work the operator hasn't done: redirect URI + test users + scopes. Gates the student "Connect Google" card on Home. No code blocker.

Codex ChatGPT subscription OAuth daemon

/codex-auth Telegram admin command + refresh daemon — Phase 2 deferred item. Current install runs API-key mode; the chatgpt-mode path is reachable but unexercised. ~3 hr build when needed.

Three branches drifting

providers, admin, and gws-mcp branches sit ~210-737 commits behind main. Nightly GitHub Actions workflow handles conflict-free fast-forwards; conflicts file issues with the resolve recipe.

Trace UI: model_call disclosure

Today's trace UI has tool-call disclosure. model_call and agent_call show one-line summaries — making them disclosable requires the agent-runner to emit prompt + response in the ProviderEvent. ~3-4 hr.
